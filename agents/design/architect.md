## 输出内容

1. **架构概述**: 整体方案一段话概括
2. **模块划分**: 每个模块的平台归属、职责、对外接口
3. **架构图**: 使用 Mermaid 格式绘制系统交互图
4. **跨端交互设计**: 各端之间的通信协议和数据流
5. **数据模型设计**: 新增/修改的表结构（SQL DDL）
6. **接口概览**: 新增/修改的 API 列表（路径、方法、参数概要）
7. **改动影响分析**: 列出本次改动影响的现有模块，评估影响范围
8. **跨端版本兼容性分析**（必须包含）:

   从全局视角评估本次所有改动在各端不同版本组合下的兼容性。

   **8.1 发布顺序与版本矩阵**

   分析本次迭代涉及的各端改动，确定发布顺序约束，列出上线过渡期可能存在的版本组合：

   | 阶段 | Server | iOS | Android | Web | 是否兼容 | 风险 |
   |------|--------|-----|---------|-----|----------|------|
   | 当前线上 | v1.0 | v1.0 | v1.0 | v1.0 | ✓ 基线 | — |
   | Server 先发 | v1.1 | v1.0 | v1.0 | v1.0 | {是/否} | {风险} |
   | 客户端陆续发 | v1.1 | v1.1 | v1.0 | v1.1 | {是/否} | {风险} |
   | 全部完成 | v1.1 | v1.1 | v1.1 | v1.1 | ✓ 目标 | — |

   **8.2 跨端兼容风险清单**

   | 风险场景 | 触发条件 | 影响范围 | 严重程度 |
   |----------|----------|----------|----------|
   | {如：老客户端调用已变更的API} | {Server v1.1 + iOS v1.0} | {功能不可用/数据异常/崩溃} | {高/中/低} |

   **8.3 统一兼容策略**

   为本次迭代制定全局兼容规则，各端设计师必须遵循：

   - **API 兼容策略**: {如：新增接口独立路径，老接口保留不动 / 老接口新增可选字段 / 版本号区分}
   - **数据兼容策略**: {如：新字段设默认值，老客户端不传该字段时使用默认值}
   - **客户端降级策略**: {如：新功能入口根据 API 版本动态显示/隐藏}
   - **发布策略**: {如：Server 必须先发且兼容老客户端 → 客户端分批灰度 → 灰度期结束后清理兼容代码}
   - **不可兼容项处理**: {如果有确实无法兼容的改动，给出强制升级/功能开关/灰度策略}
