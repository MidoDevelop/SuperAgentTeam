## 审核步骤

1. **架构一致性**: 各端方案是否与总体架构一致
2. **最小改动检查**: 是否有不必要的改动？能否用更小的改动实现？
3. **接口对齐**: 服务端 API 定义与客户端调用是否完全一致
4. **向后兼容**: 改动是否会破坏现有功能
5. **跨端版本兼容性审核**:
   - 架构设计中的「跨端版本兼容性分析」是否完整覆盖了所有版本组合场景
   - 「统一兼容策略」是否合理、可执行
   - 各端方案中的「老版本兼容性评估」是否与统一策略一致
   - 各端兼容方案之间是否存在矛盾（如 Server 方案假设客户端会处理某个字段，但客户端方案中未体现）
   - 每个标注"影响老版本"的改动项是否都给出了可行的兼容方案
   - **缺少兼容性评估、各端策略不一致、或兼容方案不可行的，标记为 blocker**
6. **风险检查**: 方案是否引入新的技术风险
7. **可执行性**: 方案是否足够详细，开发者能否直接按方案编码

## 输出内容

1. **审核结论**: approved / approved_with_comments / revision_required / rejected
2. **问题清单**: 每个问题标注级别（blocker / major / minor / suggestion）
3. **各端审核意见**: 每个平台的方案逐一评价
4. **修改建议**: 如果 revision_required，给出具体修改方向

将结果写入指定的输出文件。
