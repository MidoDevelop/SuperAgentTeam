## 输出内容

1. **改动清单**: 需要新增和修改的文件列表，每个文件标注改动类型（新增/修改）和改动描述
2. **详细方案**: 每个改动点的实现方式（类/函数签名、关键逻辑伪代码）
3. **接口实现**: 本端需要实现的 API 或调用的 API 的详细定义
4. **数据层变更**: 本端需要的数据库/存储变更
5. **遵循的编码规范**: 基于扫描结果总结的本端编码约定，开发时必须遵循
6. **改动影响**: 本次改动可能影响的现有功能，需要回归测试的范围
7. **老版本兼容性评估**（必须包含）:
   对每个改动点（新增和修改），逐一评估对老版本用户的影响：

   | 改动项 | 类型 | 影响老版本 | 影响描述 | 兼容方案 |
   |--------|------|-----------|----------|----------|
   | {改动项} | 新增/修改 | 是/否 | {如有影响，说明什么场景下老版本用户会受影响} | {必须给出兼容方案} |

   **兼容性原则**:
   - 新增 API：老版本客户端不调用即不影响，但需确认是否有强制升级逻辑
   - 修改 API：必须向后兼容，新增字段设为可选，不得删除或重命名已有字段
   - 数据库变更：只允许 ADD COLUMN（带默认值），不得删除或修改已有字段
   - 客户端 UI 变更：考虑服务端 API 未同步升级时的降级展示
   - 如果确实无法兼容，必须给出版本过渡方案（如灰度发布、强制升级、功能开关等）

将结果写入指定的输出文件。
