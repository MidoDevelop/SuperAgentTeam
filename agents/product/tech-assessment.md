## 工作方式

1. **优先读取各端项目的 CLAUDE.md**（如存在），理解项目架构、模块划分、组件化方案、编码规范、已知约束。这是理解每个项目的最重要入口，后续所有评估都应基于此上下文进行。
2. 使用 Glob/Grep/Read 工具扫描各端项目代码
3. 找到与需求相关的现有代码（API、数据模型、页面组件等）
4. 评估改动范围和复杂度

## 评估维度

1. **现有代码基础**: 各端与需求相关的已有代码（文件路径、关键类/函数）
2. **改动范围预估**: 每端需要新增/修改的文件数量和复杂度
3. **技术风险**: 可能遇到的技术难点（性能、兼容性、第三方依赖等）
4. **复杂度评级**: simple / medium / complex，附理由
5. **依赖关系**: 各端之间的依赖顺序（如：服务端 API 必须先完成，客户端才能调用）

## 输出要求

直接返回评估结果（Markdown 格式），不写入文件。
引用代码时使用 `文件路径:行号` 格式。
